# Zero Net Export 0.2.1 release validation

Date: 2026-06-30
Status: released through GitHub/HACS; live validation mostly complete

## Scope

Release `0.2.1` validates the second Home Assistant application workflow slice
through the approved release-management path. All live Home Assistant changes
must flow through GitHub release publication and HACS install/update; no direct
Home Assistant custom-component file backend deployment is approved.

## Candidate contents

- In-app navigation for Overview, Sources, Managed Devices, Controls, Runtime,
  Diagnostics, and Settings.
- App-side managed-device enable/disable actions scoped by config entry and
  managed-device key.
- Strong-confirmation managed-device removal using `REMOVE FROM ZNE`, preserving
  the original/source Home Assistant device/entity.
- App-side edits for existing controller enabled, mode, target export, deadband,
  and battery reserve control paths.

### Managed-device remove live attempt

Executed after approval on 2026-06-30:

- Pre-check: `sensor.zero_net_export_managed_devices_count=0`.
- Pre-check: `light.7th=off`, friendly name `7th`.
- Added a temporary disabled fixed managed load through the supported Home
  Assistant subentry flow:
  - name: `7th validation load`
  - entity: `light.7th`
  - nominal power: `10 W`
  - priority: `999`
  - enabled: `false`
- Add result: flow aborted with `managed_device_saved`.
- Managed count after add: `1`.
- Confirmed managed-device key: `7th_validation_load`.
- Confirmed `light.7th` still existed and remained `off`.

Installed-app removal attempt:

- Browser was force-reloaded with cache disabled to ensure the installed
  `0.2.1` tabbed app was active.
- Browser proof before removal:
  `validation/artifacts/zne-0.2.1-managed-remove-before.png`.
- The app Managed Devices tab showed the temporary managed record in the fleet
  summary.
- The browser filled managed-device key `7th_validation_load`, typed
  `REMOVE FROM ZNE`, and clicked `Remove from ZNE only`.
- Result: the installed `0.2.1` app did **not** call the remove service
  successfully. It displayed `Enter a managed-device key first.`
- Browser proof of the failed installed-app removal:
  `validation/artifacts/zne-0.2.1-managed-remove-after.png`.
- Root cause found in repo source: `_handleAction()` set busy state and called
  `_render()` before reading the managed-device key and confirmation fields, so
  the re-render cleared those inputs before the remove branch read them.

Cleanup and preservation proof:

- Removed the temporary managed record through the guarded backend service with
  `entry_id=01KW6SCBRFGG7Q641C09BXTF7N`,
  `device_key=7th_validation_load`, and `confirm=true`.
- Managed count after cleanup: `0`.
- `light.7th` remained present and `off`.

Conclusion:

- Do not claim the installed `0.2.1` app strong-confirmation removal path is
  live-validated.
- The backend guarded remove service preserved the original/source Home
  Assistant entity during cleanup.
- A repo fix now captures the managed-device key and confirmation before the
  app busy-state render; that fix is repo-validated and awaits a GitHub/HACS
  release before live app-removal validation can be retried.
